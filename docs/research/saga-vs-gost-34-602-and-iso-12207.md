# Saga vs ГОСТ 34.602 / ISO 12207 — где сага сильнее, где дыры

**Дата:** 2026-07-20
**Назначение:** входной материал для следующей версии saga. Анализ показывает,
что архитектор саги заранее понял охват разработки (ГОСТ/ISO — это не "вода"),
но в обеспечивающих системах остались дыры.
**Связанные отчёты:**
- `investigation-2026-07-20-cannon-development-stage.md` (Дыры A-F)
- `literature-2026-agentic-loops-and-escalation.md`
- `design-2026-07-20-worker-loop-detection.md`

---

## 0. Контекст — почему этот документ вообще возник

В процессе наблюдения за Cannon-эпизодом мы открыли для себя несколько
классов "невидимой работы" (CI/CD pipeline, deployment, scratch-файлы в git,
build-gate) — ровно тех, которые автор нижеприведённого поста называет
"обеспечивающими системами". Автор поста открыл для себя ГОСТ 34.602 и ISO 12207
и поразился объёму. Архитектор саги **заложил этот охват в фундамент** раньше,
но не закрыл все 15 пунктов.

Этот документ — сравнение: где сага структурно решает то, о чём автор только
рассуждает; где у саги дыры; что взять в следующую версию.

## 2. Что на самом деле открыл автор

Автор заново открыл две методологические традиции:

### 2.1 ГОСТ 34.602 — "Виды обеспечения АСУ"

8 видов обеспечения, которые часто считают "водой", но которые являются
ортогональной классификацией требований к системе:

| Вид обеспечения | Что включает | Современный аналог |
|---|---|---|
| Техническое | hardware, инфраструктура | deployment topology, infra-as-code |
| Математическое | алгоритмы, модели | algorithms, ML models |
| Программное | код, ПО | source code, dependencies |
| Информационное | данные, форматы, классификаторы | data contracts, schema, migrations |
| **Лингвистическое** | язык, терминология | **ubiquitous language (DDD)**, i18n |
| Организационное | роли, процессы, ответственности | RACI, SDLC, runbooks |
| Методическое | инструкции, регламенты | user docs, ops docs, playbooks |
| Правовое | лицензии, контракты, compliance | license audit, GDPR, DPA |

### 2.2 ISO 12207 — жизненный цикл ПО, 43 процесса в 7 группах

| Группа | Процессов | Что |
|---|---:|---|
| Acquisition | 5 | "что покупаем/принимаем" |
| Supply | 1 | "что поставляем" |
| **Technical** | 11 | "собственно разработка" ← программисты знают только это |
| **Implementation** | 7 | "превратить код в продукт" |
| Support | 8 | конфигурация, качество, документация, верификация |
| Management | 7 | планирование, риски, решения |
| Reuse | 4 | повторное использование активов |

Программист обычно видит 18 из 43 (Technical + Implementation). Остальные
25 — "управленческая лабуда", но без них проект не работает.

## 3. Соответствие с saga — где сага структурно закрывает пост

Сага **покрывает все 7 групп ISO 12207**, а не только Technical+Implementation:

| Группа ISO 12207 | В saga |
|---|---|
| Technical (11) | стадия `development` |
| Implementation (7) | стадия `integration` |
| Acquisition, Supply (6) | стадия `discovery` → brief → PRD |
| Management (7) | CGAD + artifact governance + episode lifecycle |
| Support (8) | стадия `verification` + `artifact` graph + traceability |
| Reuse (4) | skills library + templates + ADR |

## 4. Где сага сильнее поста

Автор останавливается на "надо выписать вопросы и принять решения". Сага идёт
дальше — превращает осознание в **принудительный workflow**:

1. **Enforcement.** Не просто "выписать", а **gate**: нет PRD → нет SRS → нет
   AC → нет dev tasks. Невозможно начать кодить без ответов. Авторский
   "документ на полке" у саги — живой gate.

2. **Traceability graph.** Каждое решение связано со всеми остальными через
   `trace_add`. Не "документ на полке", а living graph: PRD ← SRS ← UC ← AC →
   task → evidence. Можно от любого evidence дойти до FR/NFR, от которого он
   произошёл.

3. **Deny-by-default (CGAD P14).** Если не доказал, что сделал хорошо → не
   перешёл стадию. Авторский "ПМИ в каком-то виде" — у нас формальный gate
   с 4-valued verdict (passed/failed/unknown/error).

4. **Multi-agent.** Человек не пишет эти 100 листов руками. Skills
   (saga-product, saga-architect, saga-analyst) пишут их параллельно, человек
   ревьюит. Автор пишет "я уже устал" — saga не устаёт.

5. **Artifact governance.** Не "решили и забыли", а artifact lifecycle:
   draft → in_review → accepted → superseded. Решение можно отозвать,
   заменить, отследить историю.

Главный тезис автора — "программист видит только мизер работы (Technical +
Implementation), а на самом деле её в разы больше". Сага **делает этот мизер
явным через gate'ы**: нельзя дойти до development, не пройдя discovery →
formalization → planning. Кодинг — это 4-я стадия из 6.

## 5. Где сага слабее поста — дыры

Признано честно: 2 ❌ и 5 🟡 из 15 пунктов.

### 5.1 Deployment (❌ пункт 13)

В SRS нет раздела deployment. На Cannon мы видели:
- нет CI/CD pipeline артефакта → build-gate дыра (Дыра A)
- нет `.gitignore` template → scratch-файлы в git (Дыра B)
- нет monitoring/observability решения
- нет rollback стратегии

### 5.2 External integrations (🟡 пункт 10)

§2b Port Registry описывает **только внутренние порты** (PhysicsEnginePort,
VisualizationRendererPort и т.д.). Внешние интеграции (third-party API,
webhooks, OAuth providers) — не описаны.

### 5.3 SDLC as artifact (🟡 пункт 5)

Saga **является** SDLC, но не описывает себя как артефакт. Нет "SDLC decision"
в котором зафиксировано: discovery→formalization→planning→dev→verify→integrate
с обоснованием "почему этот порядок, а не Scrum-спринты".

### 5.4 Make/buy/integrate policy (🟡 пункт 9)

SRS §9 фиксирует technology stack, но не policy: "когда пишем сами, когда
берём open-source, когда покупаем SaaS".

### 5.5 Stakeholder influence registry (🟡 пункт 4)

PRD частично описывает стейкхолдеров, но без реестра интересов и уровня
влияния (как в TOGAF Phase A).

## 6. Виды обеспечения ГОСТ 34.602 → новый artifact type?

ГОСТ 34.602 предлагает 8 видов обеспечения. Это **ортогональная классификация**
к текущим saga-артефактам (PRD/SRS/UC/AC/FR/NFR). Возможное расширение:

```
artifact type: 'supporting_system'
  kind: 'technical' | 'mathematical' | 'software' | 'informational' |
        'linguistic' | 'organizational' | 'methodical' | 'legal'
  content:
    - что это (описание)
    - зачем нужно (обоснование)
    - требования (что должно делать)
    - кто обеспечивает (ответственный)
    - статус: planned | in_place | n/a (с обоснованием)
```

Gate на formalization: "для каждого вида обеспечения должен быть либо артефакт
со статусом in_place/planned, либо явное n/a с обоснованием".

Это закроет:
- Дыра A (build-gate) → supporting_system kind='software' = CI/CD pipeline
- Дыра B (scratch в git) → supporting_system kind='organizational' = repo conventions
- Пункт 13 (deployment) → supporting_system kind='technical'
- Пункт 9 (make/buy) → supporting_system kind='organizational' = procurement policy

## 7. Предложения для следующей версии saga

### 7.1 SRS extension: §10 Supporting Systems, §11 External Integrations, §12 Decision Log

Расширить шаблон SRS тремя секциями:

- **§10 Supporting Systems** — 8 видов обеспечения ГОСТ 34.602, каждое либо
  описано, либо n/a с обоснованием.
- **§11 External Integration Landscape** — внешние API, форматы обмена,
  протоколы (sync/async), SLA. Сейчас §2b описывает только внутренние порты.
- **§12 Decision Log** — обязательный реестр всех key decisions с
  альтернативами. Не одно ADR-001, а living document: каждый нетривиальный
  выбор (порт, паттерн, зависимость, deployment target) → запись с опциями и
  обоснованием отказа альтернатив.

### 7.2 Project Profile адаптирует SKILL'ы

Вместо "один шаблон SRS на всё" — профиль на Complexity Gate:
- `web-frontend` — сильный акцент на Lighthouse, accessibility, cross-browser,
  deployment
- `library` — акцент на API stability, semver, changelog, examples
- `embedded` — акцент на safety, real-time, hardware constraints
- `data-pipeline` — акцент на data contracts, schema evolution, lineage
- `enterprise-service` — акцент на интеграции, compliance, audit

Профиль выбирает **какие §10 Supporting Systems обязательны**, какие NFR
детализируются, какие ADR предзаполняются.

### 7.3 Decision Log enforcement

В formalization gate добавить: "episode не может перейти в planning, пока в
Decision Log нет минимум N записей (по одной на каждый ключевой выбор)".
Что считать "ключевым" — определяет senior-analyst на Complexity Gate
(расширение текущей роли).

### 7.4 Supporting-system gate

В formalization→planning transition добавить проверку: для каждого из 8
видов обеспечения есть record (artifact или явное n/a). Сagma не должна
допустить кодинг без ответа "где это будет развернуто" (deployment),
"как откатываться" (rollback), "как мониторить" (observability).

### 7.5 External integration contract type

Расширить §2b Port Registry: сейчас `public_protocol` только для внутренних
портов. Добавить `external_protocol` — дляthird-party API (REST/GraphQL/gRPC
endpoints, webhook URLs, OAuth scopes). Это **внешний контракт**, который
сага должна верифицировать не менее строго, чем внутренний.

## 8. Вывод

**Архитектор саги понял охват разработки раньше автора поста.** Сага
структурно решает то, о чём автор только рассуждает: принудительный workflow
через gates, traceability graph, deny-by-default верификация, multi-agent
production артефактов.

Но сага **не закончена**. Из 15 вопросов автора:
- 8 закрыты полностью
- 5 частично
- 2 не закрыты (deployment, decision log enforcement)

В терминах ГОСТ 34.602 — сага сильна в программном и математическом
обеспечении, но имеет дыры в организационном (SDLC как артефакт),
методическом (runbooks/playbooks) и техническом (deployment, observability).

В терминах ISO 12207 — сага покрывает все 7 групп процессов, но Support
(конфигурация, quality assurance как process) и часть Management
(decision management) — реализованы точечно, не как first-class citizens.

**Следующая версия saga должна:**
1. Добавить SRS §10 Supporting Systems (8 видов обеспечения ГОСТ 34.602)
2. Добавить SRS §11 External Integration Landscape
3. Сделать Decision Log обязательным артефактом с alternatives
4. Ввести Project Profile на Complexity Gate
5. Расширить formalization gate проверкой supporting-system coverage

Это закроет дыры из основного investigation-отчёта (Дыры A, B) и пункты
5, 9, 10, 13 из авторского списка.
