**Связанные документы:** `docs/tracker-view-backlog.md` (backlog + матрица приоритетов).

Два независимых дизайна, объединённых в один документ, поскольку оба касаются

# Раздел 1: BR (бизнес-требования) как тип артефакта

## 1.1 Контекст и проблема

В схеме saga-mcp сейчас 9 типов артефактов
(`PRD/SRS/UC/AC/FR/NFR/decision/theme/brief`). Канонический список зафиксирован в
**4 местах** (комментарий в `src/schema.ts:217-221` явно требует синхронизации):
1. `src/schema.ts` — SQL CHECK constraint `artifacts.type IN (...)` (строка 139).
2. `src/types.ts` — TypeScript union `ArtifactType` (строка 96-99).
3. `src/tools/artifacts.ts` — `ARTIFACT_TYPES` const (строка ~29) + `ArtifactTypeSchema`
   zod-enum (`src/schema.ts:224`).
4. `tracker-view/tracker-view.mjs` — `TYPE_COLORS`/`TYPE_LABEL` (строки 47-54).

BR (Business Requirements) отсутствует. PRD = Product Requirements, но
бизнес-требования — отдельная сущность **выше** PRD в классической пирамиде:

```
BR (почему: метрики, стейкхолдеры, рамки)
 └─ PRD (что: продукт, фичи, пользовательские истории)
     ├─ FR / NFR / RULE   ← живут в PRD с ADR-014 (раньше были в SRS)
     ├─ UC → AC           ← UC/AC пишутся по PRD ДО SRS
     └─ SRS (ПОСЛЕ AC)    ← архитектор видит замороженные AC + complexity
         └─ DECOMP §D     ← per-AC map: files/functions/types/conflict_keys/ac_kind
              ↘ (implements) → dev-tasks (planner копирует §D2 дословно)
```

> **Note (ADR-014, 2026-07-20):** SRS перенесён с параллельного с UC положения
> на позицию ПОСЛЕ AC. FR/NFR/RULE переехали из SRS в PRD. SRS стал чисто
> архитектурным артефактом + содержит §D DECOMP, по которому planner создаёт
> задачи (dumb copier).

Сейчас «почему» либо вообще не зафиксировано, либо вмазано в PRD/decision.
Гипотеза пользователя: ввести BR как parent для PRD.

## 1.2 Варианты

### Вариант A — Новый тип `BR` (полноценная сущность)

Добавить `'BR'` в канонический список артефактов. BR → parent для PRD
(`parent_artifact_id`). Содержимое .md: цели, метрики успеха (KPI),
стейкхолдеры, рамки (in/out of scope), риски.

**Плюсы:**
- Чистая модель; явная пирамида BR→PRD→SRS→AC.
- Отдельная иконка/цвет в дереве tracker-view; отдельный реестр `?registry=BR`.
- Регулярный путь расширения (theme/brief уже добавлялись так же).

**Минусы:**
- Schema change в горячей БД (см. §1.4 — технически сложно).
- Надо мигрировать существующие PRD (прикрепить к новому BR-parent) — рискованно.

### Вариант B — Reuse существующего `decision`/`brief` для бизнес-контекста

Не добавлять новый тип. Использовать `decision` (discovery-артефакт) или `brief`
как контейнер бизнес-контекста, parent для PRD. Например: PRD `derived_from`
brief, а brief держит KPI/стейкхолдеров.

**Плюсы:**
- 0 schema changes — безопасно, можно сделать сейчас.
- brief уже валидируется (`validateBrief`, 12 секций) — частично покрывает бизнес-контекст.

**Минусы:**
- Семантическая перегрузка: brief = discovery-output, не «бизнес-требования».
  Смешивание ролей запутает пользователей и агентов.
- Нет отдельной иконки/реестра; бизнес-контекст растворится среди discovery-брифов.

### Вариант C — Subtype через `metadata.kind` (без нового типа)

Оставить PRD, но добавить `metadata.business = true` и группу полей (KPI,
stakeholders) в metadata PRD. UI рендерит бизнес-секцию из metadata.

**Плюсы:**
- 0 schema changes; максимально гибко.

**Минусы:**
- Слабая типизация: metadata — свободный JSON, валидации нет, конвенция хрупкая.
- Теряется иерархия BR→PRD (нет отдельного узла-родителя).
- Не попадает в реестр/дерево как отдельная сущность.

## 1.3 Матрица оценки

Критерии (1-5, как в backlog): Сложность↓, Ценность↑, Риск↓, Обратная_совместимость↑.

Итог = `Ценность + Обратная_совместимость − Сложность − Риск`.

| Вариант | Слож↓ | Цен↑ | Риск↓ | Совм↑ | Итог |
|---|---|---|---|---|---|
| **A** Новый тип `BR` | 3 | 4 | 4 | 4 | **+1** |
| **B** Reuse decision/brief | 1 | 2 | 1 | 4 | **+4** |
| **C** metadata.kind | 1 | 2 | 1 | 4 | **+4** |

## 1.4 Техническая сложность варианта A (schema change)

CHECK constraint в SQLite **нельзя просто расширить**. Процедура:

```sql
-- 1. Backup (обязательно перед любой миграцией)
.backup 'C:/Users/user/.zcode/saga-backup-<date>.db'

-- 2. Создать новую таблицу с расширенным CHECK
CREATE TABLE artifacts_new (
  ... те же колонки ...
  type TEXT NOT NULL CHECK (type IN ('PRD','SRS','UC','AC','FR','NFR',
                                     'decision','theme','brief','BR')),
  ...
);

-- 3. Скопировать данные
INSERT INTO artifacts_new SELECT * FROM artifacts;

-- 4. Атомарная замена в транзакции
BEGIN;
  DROP TABLE artifacts;
  ALTER TABLE artifacts_new RENAME TO artifacts;
  -- пересоздать индексы (idx_artifacts_*)
COMMIT;
```

Риски: блокировка записи на время миграции (saga-mcp пишет параллельно через WAL),
потеря данных при ошибке в INSERT SELECT, рассинхронизация 4 мест если забыть
одно. **Требует explicit approval + проверка backup + прогон тестов saga-mcp.**

Дополнительно (вариант A):
- `src/types.ts`: добавить `| 'BR'` в union.
- `src/tools/artifacts.ts`: `'BR'` в `ARTIFACT_TYPES`.
- `src/schema.ts`: `'BR'` в `ArtifactTypeSchema` zod-enum.
- `tracker-view`: `BR:'#color'` в `TYPE_COLORS`, `BR:'БР'` в `TYPE_LABEL`, в
  `typeOrder` массивы (строки 560, 1196), в реестр chips (строка 373).
- skills/docs: упомянуть BR в saga-product/saga-analyst SKILL.md как верхний уровень.
- Создать BR→PRD `derived_from` трассы для существующих эпизодов (migration T1.3).

## 1.5 Рекомендация

**Рекомендуется вариант A, но только после ревью пользователя** — он даёт
наибольшую ценность (чистая модель, отдельная сущность, регулярный путь), а
техническая сложность управляема при наличии backup и тестов. Варианты B/C —
«бесплатные» костыли, но накапливают технический долг.

**Порядок внедрения (если пользователь одобряет A):**
1. Backup БД + прогон `tests/` saga-mcp на зелёность.
2. Реализовать миграцию (§1.4) в транзакции с откатом при ошибке.
3. Синхронизировать 4 места (schema/types/tools/tracker-view) + skills — одним коммитом.
4. Smoke-test: создать тестовый BR, проверить дерево/реестр/kanban.
5. (Опц.) Migration T1.3: прикрепить избранные PRD к новым BR-parent вручную
   (не автоматически — пользователь решает какие PRD заслуживают BR-обёртки).

**Пока ревью не получено** — вариант B (reuse brief) как временное соглашение:
PRD с бизнес-контекстом создавать как `brief` с `derived_from` → PRD. Это не
требует кода и обратимо.

# Раздел 2: Автоматизация трассировки UC↔FR↔AC↔task

## 2.1 Контекст и проблема

Граф трассировки (`artifact_traces`) — мост между требованиями и
канбаном строителей. Три основных `link_type` проставляются **вручную**
(через `trace_add` MCP tool или saga-planner/saga-analyst skills):

| link_type | семантика | кто проставляет | объём (по данным БД) |
|---|---|---|---|
| `derived_from` | FR/NFR/RULE→PRD (с ADR-014), SRS→PRD, UC→PRD, AC→UC, AC→FR/NFR (в PRD) | saga-product (FR/NFR/RULE), saga-analyst (UC/AC), saga-architect (SRS) | ~80+ трасс |
| `covers` | UC→FR (UC покрывает FR) | saga-analyst вручную | ~95+ трасс |
| `implements` | AC→dev-task | saga-planner при создании dev-задач (по SRS §D2 — dumb copier с ADR-014) | ~100+ трасс |

**Боль (по дереву артефактов tracker-view):**
- AC без `implements`-трассы — «висячий» критерий, не реализован. Видно только
  глазами при обходе дерева (granite: 76 артефактов).
- AC без `derived_from`→FR — критерий без прослеживаемости к функциональному
  требованию. Невозможно понять «для какого FR эта AC».
- UC без `covers`→FR — use case без покрытия функциональными требованиями.
- Ошибки в кодах: AC-1 может ссылаться на несуществующий FR-1, если FR удалён.

## 2.2 Анализ现状 (по реальным данным БД на 2026-07-04)

Изучены 280+ трасс через `trace_list`. Подтверждённые паттерны:

### Паттерн P1: парные `derived_from` AC→UC + AC→FR

AC почти всегда имеет **две** `derived_from` трассы: к UC и к FR. Пример (REQ-002,
strings-utils):
```
AC-1 (capitalize) → derived_from → UC-1 (применяет capitalize)
AC-1 (capitalize) → derived_from → FR-1 (capitalize(s))
AC-1 → derived_from → FR-4 (чистые функции)   ← кросс-связь
```
В deposit-calc: AC-1 → UC-1 + FR-2 + FR-6 (3 цели). То есть AC→FR часто
**множественный**. Это затрудняет простую авто-связь по коду AC-N→FR-N.

### Паттерн P2: `implements` AC→task, task title повторяет AC

Трассы implements: source_code `AC-1`, target_title часто буквально
`"AC-1: <перефразированный AC>"`. Это видно в calc-epic, strings-epic, granite-epic:
```
AC-1 (source) → implements → task "AC-1: implement capitalize(s) body"
AC-2 (source) → implements → task "AC-2: implement reverse(s) body"
```
**Гипотеза:** saga-planner создаёт dev-task с title `AC-<N>: ...` и сразу
проставляет `implements`. Это уже частично автоматизировано в planner-skill.
Пробелы возникают, когда planner пропускает шаг или AC создаётся позже task.

### Паттерн P3: `covers` UC→FR, множественный

UC-1 (calc: сложение) → covers → FR-1, FR-4, FR-5 (3 FR). UC-4 (granite:
simulation step) → covers → FR-4..FR-7 (4 FR). Связь семантическая (какие FR
участвуют в сценарии UC), **не** выводится из кода автоматически.

### Паттерн P4: пробелы в implements (видны по статусам)

Geosophia (epic REQ-CC): AC-CC-001..008, большинство target_status=`blocked` —
dev-задачи есть но заблокированы. TestLasGPU: AC-1..AC-9, часть без implements
вообще. Это **реальные пробелы покрытия**, которые coverage-matrix должна
показывать явно.

## 2.3 Варианты автоматизации

### (a) Auto-suggest трасс при создании артефакта

При `artifact_create` типа AC: предложить `derived_from`→FR по совпадению номера
в коде (AC-N → FR-N в том же эпизоде). При создании task с title `AC-<N>:` —
предложить `implements`→AC.

**Реализация:** серверная подсказка (не авто-запись) — MCP tool `trace_suggest`
возвращает кандидатов, пользователь/планировщик подтверждает `trace_add`.

**Плюсы:** снижает ручную работу на ~60% (паттерн P1, P2).
**Минусы:** множественные связи (P1: AC→несколько FR) не угадываются; нужен
контекст названия. Риск «загрязнения» графа при авто-записи без подтверждения.

### (b) UI-индикаторы пробелов трассировки

В дереве артефактов tracker-view рядом с AC:
- ⚠ оранжевый, если нет `implements` → task (AC не реализована).
- ⚠ серый, если нет `derived_from` → FR (AC не отслежена к требованию).
- Аналогично для UC: нет `covers` → FR.

**Реализация:** в `renderNode` (tracker-view.mjs) проверять `tracesBySource[art.id]`
на наличие нужных link_type. Read-only, аддитивно, риск 1/5.

**Плюсы:** мгновенная визуальная диагностика покрытия; 0 изменений данных.
**Минусы:** только индикация, не автоматизация создания трасс.

### (c) Coverage-matrix вкладка (РЕАЛИЗОВАНО в эту ночь — см. backlog T2.1)

`/?project=N&tab=coverage` — таблица AC × {implements, verified_by}: AC без
трасс — красным, с трассами — зелёным. Read-only, переиспользует SQL из
`handleArtifactCoverage` (src/tools/artifacts.ts:387).

**Плюсы:** агрегированный обзор покрытия всего проекта; экспортируемая картинка.
**Минусы:** плоская таблица, не показывает иерархию UC→FR→AC.

### (d) Парсинг таблиц трасс из .md

Многие .md артефактов содержат таблицы трасс в markdown
(`| AC | FR | task |`). Парсер читает такие таблицы и создаёт/обновляет
`artifact_traces` для рассинхронизированных.

**Реализация:** MCP tool `trace_import_from_md` (artifact_id) → парсит .md,
сравнивает с существующими трассами, возвращает diff (create/delete).

**Плюсы:** .md как single source of truth; трассы самосинхронизируются с документом.
**Минусы:** хрупкий парсинг (формат таблиц может варьироваться); неоднозначность
имен FR-1 vs FR-001; риск ложных трасс. Высокая сложность (8ч), нужен прототип.

## 2.5 Рекомендация

**Сделано в эту ночь:** вариант **(c)** coverage-matrix — наилучшее соотношение
ценность/риск, read-only, решает главную боль («какие AC не реализованы»).

**Следующий sprint (после ревью):**
1. Вариант **(b)** UI-индикаторы — аддитивно к дереву, риск 1/5, мгновенная
   польза. Рекомендуется к реализации сразу.
2. Вариант **(a)** auto-suggest — только как **suggest** (не авто-запись),
   интегрировать в saga-planner/saga-analyst skills. Прототип на одном эпизоде.

**Прототипы (исследовательские):** вариант **(d)** парсинг .md — отдельная ветка,
сначала на одном эпизоде (REQ-001 calc) для валидации формата таблиц.

## 2.6 Что НЕ автоматизировать (вывод из анализа)

- `covers` UC→FR (паттерн P3) — семантическая связь, не выводится из кода.
  Оставить ручным через saga-analyst.
- Множественные `derived_from` AC→FR (AC-1 → FR-1 + FR-4) — авто-предложение по
  номеру даст только FR-1, кросс-связи потеряются. Только suggest, не авто-запись.
