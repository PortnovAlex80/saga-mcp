## Краткое описание идей

### Идея 1 — BR (бизнес-требования) как отдельный тип артефакта

Сейчас в схеме saga-mcp 9 типов артефактов: `PRD/SRS/UC/AC/FR/NFR/decision/theme/brief`.
BR отсутствует. PRD = Product Requirements, но бизнес-требования («почему мы это делаем»,
метрики успеха, стейкхолдеры, рамки) — отдельная сущность **выше** PRD в классической
пирамиде (BR → PRD → SRS/UC/FR → AC). Гипотеза: ввести BR как тип артефакта, parent для
PRD. Это **schema change в saga-mcp TypeScript** — затрагивает `src/schema.ts` (CHECK
constraint), `src/types.ts` (ArtifactType union), `src/tools/artifacts.ts`
(ARTIFACT_TYPES const, валидаторы), tracker-view (TYPE_COLORS/TYPE_LABEL), skills/docs.

### Идея 2 — Автоматизация трассировки UC↔FR↔AC↔task

В дереве артефактов видно, что между UC и FR/AC пользователь вручную проставляет
`derived_from`/`covers`. Трассировка хрупкая, таблицы трасс сложно содержать. Нужен
глубокий анализ: где сейчас ручная работа, что можно автоматизировать (auto-link
FR↔AC по коду AC-N→FR-N, подсказки в UI «эта AC не имеет implements-трассы»,
coverage-matrix как отдельная вкладка, импорт трасс из таблиц в .md). Это
исследовательская задача — нужен дизайн-документ с вариантами и матрицей, а не сразу код.

### Идея 3 — Write-инструменты: создание проекта/эпика/артефакта из GUI

Сейчас tracker-view почти везде read-only, кроме `/api/artifact/save`. Нужны
write-инструменты:
- Создать проект (форма: name, description) → `project_create`.
- Создать эпик в проекте (форма: name, branch) → `epic_create`.
- Создать артефакт (форма: type, code, title, epic_id) → `artifact_create` + .md файл
  по конвенции `docs/requirements/REQ-NNN-slug/0X-type.md`.
- Построить дерево документов из шаблона («Создать эпизод REQ-NNN» → PRD+SRS+UC+AC+FR скелет).

Критерии (1-5):
- **Сложность** — инженерные усилия (минимизировать). 1 = мелкая правка, 5 = мульти-файловая миграция.
- **Ценность** — польза для пользователя (максимизировать). 1 = косметика, 5 = меняет рабочий процесс.
- **Риск** — вероятность сломать существующее (минимизировать). 1 = изолированная аддитивная фича, 5 = schema change в горячей БД.
- **Зависимость** — нужны ли другие этапы раньше (0 = независимо, иначе id этапа).

### Что ОПАСНО делать автономно (без ревью)

| Задача | Почему опасно |
|---|---|
| **1a — BR schema change** | Меняет CHECK constraint `artifacts.type` в горячей БД. CHECK в SQLite нельзя «просто расширить» — нужна пересборка таблицы (`CREATE TABLE _new … INSERT SELECT … DROP … RENAME`), что блокирует запись и рискует потерей данных при ошибке. Кроме того, синхронизация 4 мест (schema.ts, types.ts, artifacts.ts ARTIFACT_TYPES, tracker-view TYPE_COLORS) — легко рассинхронизировать, и валидация начнёт отбрасывать легитимные артефакты. Требует explicit approval + backup БД. |
| **1b — BR migration** | Перенос существующих PRD под нового BR-parent меняет `parent_artifact_id` у рабочих артефактов (granite, Geosophia). Любая ошибка ломает дерево, которое пользователь только что нахвалил. |
| **3c/3d — создание артефакта/мастер эпизода** | Нужна конвенция путей `docs/requirements/REQ-NNN-slug/0X-type.md`, выбор репозитория (PROJECT_REPO_MAP), и запись .md файлов в чужие репо (Stone, geosophia…). Неверный путь = мусорные файлы в рабочих репозиториях. Без зафиксированной пользователем конвенции — риск. |
| **2c — auto-suggest трасс** | Автоматическое создание `trace_add` без подтверждения может навесить неверные связи и «загрязнить» граф трассировки, который потом придётся чистить вручную. |
